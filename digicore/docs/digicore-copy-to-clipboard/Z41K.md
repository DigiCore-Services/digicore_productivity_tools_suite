# Implementation Plan - Phase 52: "One-Click" Corpus Generation Utility

Based on architectural requirements, the Corpus Generation Utility will be implemented as a robust, decoupled sub-application strictly adhering to Hexagonal Architecture, SOLID, and SRP principles. 

This ensures extensibility (e.g., adding cloud corpus backup, alternative baseline generators) and aligns with our configuration-first ecosystem strategy.

## Proposed Changes

### 1. Domain Layer (`digicore-core`)

#### [NEW] `src/domain/ports/corpus.rs`
Define the interfaces for the sub-application boundaries to decouple the business logic from filesystem or OCR implementations.
- `CorpusStoragePort`: Defines the contract for persisting the captured image. (`fn save_image(&self, data: &[u8], filename: &str) -> Result<PathBuf>;`)
- `CorpusBaselinePort`: Defines the contract for generating the regression baseline. (`fn generate_baseline(&self, image_path: &PathBuffer) -> Result<PathBuf>;`)

#### [NEW] `src/domain/value_objects/corpus.rs`
- `CorpusConfig`: Configuration struct allowing dynamic toggle and hotkey mapping.
    ```rust
    pub struct CorpusConfig {
        pub enabled: bool,
        pub shortcut_modifiers: u16, // e.g. Ctrl|Alt|Shift bitmask
        pub shortcut_key: u16,       // e.g. 'S'
        pub output_dir: String,
        pub snapshot_dir: String,
    }
    ```

### 2. Adapters (`digicore-text-expander`)

#### [NEW] `src/adapters/corpus/filesystem_storage.rs`
- `FileSystemCorpusStorageAdapter`: Implements `CorpusStoragePort`. Handles creating directories and writing the binary image data to disk (`docs/sample-ocr-images`).

#### [NEW] `src/adapters/corpus/ocr_baseline.rs`
- `OcrBaselineAdapter`: Implements `CorpusBaselinePort`. Injects the existing `WindowsNativeOcrAdapter`, runs extraction on the new image, and formats/saves the output as an `insta` `.snap` file in `tests/snapshots`.

### 3. Application Layer (`digicore-text-expander`)

#### [NEW] `src/application/corpus_generator.rs`
- `CorpusService`: The orchestrator core for this sub-application.
    - Maintains the dependencies (`ClipboardPort`, `CorpusStoragePort`, `CorpusBaselinePort`, `CorpusConfig`).
    - Exposes a `try_capture()` method.
    - Logic:
        1. Checks if clipboard contains an image.
        2. Generates a timestamped filename.
        3. Calls storage adapter to save the image.
        4. Calls baseline adapter to generate the `.snap`.
        5. Fires OS Toast notification ("Added to corpus: [filename]").

### 4. Integration Point (`digicore-text-expander`)

#### [MODIFY] `src/drivers/hotstring.rs`
- Inject the logic to listen for the `CorpusConfig` hotkey combo (e.g., `Ctrl+Alt+Shift+S`).
- Since we already maintain global keyboard state, we'll watch for this combo and dispatch to `corpus_generator::try_capture_global()` (which holds a static Mutex reference to the `CorpusService`).

#### [MODIFY] `src/main.rs` & `app_state.rs`
- Initialize `CorpusConfig` with defaults holding "docs/sample-ocr-images".
- Initialize `CorpusService` with concrete filesystem/OCR adapters and start the background listener hook.

## Verification Plan

### Automated/Unit Tests
- Write unit tests for `CorpusService` using mock ports.
- Ensure `CorpusService.try_capture()` returns graceful errors if the clipboard holds text instead of an image.

### Manual Verification
1. Run application.
2. Snipping-tool a region of screen.
3. Press `Ctrl+Alt+Shift+S`.
4. Verify OS Toast.
5. Verify `.png` in `docs/sample-ocr-images/`.
6. Verify `.snap` in `tests/snapshots/`.
